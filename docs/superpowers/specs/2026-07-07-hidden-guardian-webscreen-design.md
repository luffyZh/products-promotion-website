# 隐卫 Web 大屏模块设计

## 背景

当前 `SpiritualEyes.tsx` 已经完成以下内容：

- 首屏双模能力表达：`DVS` 负责识别偷拍行为，`IR` 负责探查偷拍设备。
- 两大产品形态：隐卫穹顶、隐卫随行。
- 四大核心场景：面向高密级空间和高风险行为的落地场景。
- 技术对比：普通摄像头与 `DVS` 事件相机的画面差异。

现阶段页面仍然偏“设备能力展示”，缺少一个能够承接系统级能力的模块。  
用户提供了 `web-screen.png`，希望新增一个 Web 大屏系统模块，用于展示隐卫在多点部署后的统一总览、实时告警与事件闭环能力。

## 目标

在隐卫页面中新增一个独立的 Web 大屏模块，强化“隐卫不仅是单点设备，更是可被统一接入、统一总控的完整系统”这一认知。

本模块的目标如下：

- 展示隐卫 Web 大屏系统的系统级能力。
- 将前面的“设备形态”与后面的“场景落地”连接起来。
- 强调房间态势总览、设备在线状态、实时告警流和事件处置闭环。
- 使用真实素材 `web-screen.png` 作为主视觉，而不是占位图。

## 模块定位

本模块的角色是：`态势总览中枢`。

它不是一个普通的图片展示区，也不是技术对比的一部分，而是隐卫系统化能力的承接模块。  
用户浏览到这里时，已经知道隐卫能“探设备、识行为”，也已经看到了不同设备形态。此时最自然的下一步，是告诉用户：这些设备如何在真实环境中接入统一平台，被集中查看、联动告警与闭环处置。

因此，本模块承担以下叙事职责：

- 说明隐卫具备平台化接入能力。
- 说明设备部署后可进入统一 Web 大屏。
- 说明大屏不是单纯展示，而是服务于告警联动和管理闭环。

## 插入位置

采用 `A` 方案。

### 最终位置

将 Web 大屏模块插入在：

- `两大产品形态` 模块之后；
- `四大核心场景` 模块之前。

### 页面顺序

调整后的页面叙事顺序为：

1. 首屏：隐卫核心能力认知
2. 核心能力卡片：产品价值拆解
3. 两大产品形态：部署方式与设备形态
4. Web 大屏模块：统一总控与态势总览
5. 四大核心场景：实际落地应用
6. 技术对比：普通摄像头 vs `DVS` 事件相机

### 选择原因

这个位置最顺畅，原因如下：

- 先讲设备，再讲平台，符合用户理解路径。
- 大屏模块能够自然承接“多点部署后的统一管理”这一话题。
- 放在场景之前，可以把后面的四大场景理解为平台服务的落地对象，而不只是孤立案例。
- 如果放在技术对比附近，会与“普通摄像头 vs `DVS`”的对比逻辑混在一起，削弱大屏模块的独立价值。

## 内容结构

本模块采用 `左文右图` 的布局。

### 左侧内容区

左侧用于讲清楚 Web 大屏系统的价值与能力，采用以下结构：

1. 小标签
2. 标题
3. 副标题
4. 三个能力点

### 右侧视觉区

右侧直接使用用户提供的真实素材：

- `src/assets/hidden-guard/web-screen.png`

这张图作为主视觉，展示隐卫实时监控大屏的实际界面质感。

## 文案方向

文案不走参数堆砌路线，而走“系统能力 + 使用结果”的表达方式。

### 标题建议

主标题建议为：

- `隐卫实时监控大屏`

该标题足够直接，且与截图内容一致，能迅速建立“统一总控平台”的认知。

### 副标题建议

副标题建议表达为：

- `房间态势、设备在线、实时告警、处置回流，一屏掌握。`

这句副标题要同时满足：

- 体现“总览”；
- 体现“实时性”；
- 体现“闭环管理”；
- 保持简洁，避免堆术语。

### 三个能力点建议

建议使用 3 个能力点卡片或列表项，分别表达：

1. `空间态势总览`
   - 面向房间、区域和设备点位的整体态势可视化。
   - 强调部署位置、空间分布和风险热区的统一查看。

2. `实时告警联动`
   - 设备识别结果、风险事件和告警流同步汇聚到大屏。
   - 强调“发现即上屏、告警即联动”的实时平台能力。

3. `事件闭环追踪`
   - 支持查看历史告警、事件回放、处理状态和留痕结果。
   - 强调这不是一次性提醒，而是可管理、可回溯的处置闭环。

## 视觉设计

### 总体风格

整体风格保持与隐卫页面一致，偏深蓝、科技、可信赖，但避免做成过重的纯黑大块，防止破坏页面节奏。

### 布局风格

- 左侧为文案与能力点；
- 右侧为大屏截图；
- 整体区块建议使用较高质感的容器与轻微蓝色氛围背景；
- 保持与首屏、产品卡片、技术对比区块一致的圆角、边框和阴影语言。

### 视觉重点

模块的视觉焦点应落在右侧大屏截图上，但左侧的标题层级要足够强，确保用户知道这不是普通插图，而是平台模块。

### 避免的问题

本模块视觉设计中要避免：

- 右图太小，导致大屏系统存在感不足；
- 左侧内容过多，读起来像产品说明书；
- 与后面的技术对比模块过于相似，形成视觉重复；
- 做成全屏深色重压块，导致页面节奏失衡。

## 实现边界

- 接入 `src/assets/hidden-guard/web-screen.png`。

- 同步更新：
  - `src/i18n/locales/zh-CN.json`
  - `src/i18n/locales/en.json`
  - `src/i18n/locales/ar.json`

### 本次范围外

- 不调整已有首屏标注逻辑。
- 不重构两大产品形态模块。
- 不改动四大核心场景的现有内容结构。
- 不改动“普通摄像头 vs DVS”技术对比模块的已有逻辑。

## 多语言策略

该模块需要与页面现有结构保持一致，补齐三套语言文案：

- 中文：主表达语言，重点保证语义准确和产品感。
- 英文：以产品介绍语气为主，避免直译和过度营销。
- 阿拉伯语：与英文语义保持一致，继续沿用当前页面的国际化风格。

建议新增文案键集中放在 `spiritualEyes` 命名空间下，避免与其他产品页混用。

## 推荐实现方式

在 `SpiritualEyes.tsx` 中新增一个独立 section，结构如下：

- 区块外层 section
- 左侧文案容器
- 右侧图片容器
- 左侧 3 个能力点列表

文案和图片资源都在本页面内部直接使用，不需要额外抽出公共组件。  
原因是该模块高度服务于隐卫页面本身，且当前页面文件仍处于可以接受的体量范围内。

## 验收标准

满足以下条件即可认为设计落地正确：

1. 页面新增一个独立的 Web 大屏模块。
2. 模块位置位于“产品形态”之后、“四大核心场景”之前。
3. 模块主标题能够明确表达这是隐卫的实时监控大屏系统。
4. 模块使用真实图片 `web-screen.png`，不再使用占位内容。
5. 左侧内容能够清楚说明态势总览、实时告警和事件闭环三类能力。
6. 三份国际化文案同步补齐，切换语言时结构完整。
7. 模块视觉上与现有隐卫页面风格统一，不突兀、不喧宾夺主。

## 风险与注意事项

- `web-screen.png` 信息密度较高，因此右侧图片容器必须给足展示空间，避免缩得过小。
- 若左侧文案写得过长，会与右图争抢版面，需要严格控制文案字数。
- 模块新增后，页面整体长度会增加，需注意上下区块的留白节奏是否平衡。
- 该模块应强调“系统总控能力”，避免与后面的画面对比模块发生功能语义重复。

## 结论

本次新增的 Web 大屏模块，采用 `A` 方案，作为“设备形态”与“场景落地”之间的系统能力承接层。

核心原则是：

- `先讲设备，再讲平台，再讲场景`
- `左侧讲能力，右侧讲系统感`
- `突出总控中枢，而不是简单展示一张后台截图`
